Goal, current state, target behaviour

Current state (what exists today)

  • Geometry already drives everything: plan_v##.json holds rooms as metre-rectangles + openings + fixtures; scripts/render_plan.py (Pillow) draws the styled plan; changes_v##.json drives the red-box delta overlay.
  • The review loop works but is stitched together: chat + lavish pages with hand-generated PNG hotspots; walls aren't objects — you can't select "this chunk of wall" today.
  • Sibling reference: data-qa-agent proves the target shape — React+Vite frontend, FastAPI backend, Pydantic AI agent service, Postgres, make up locally, AWS App Runner + Terraform + GitHub Actions deploy.

Target behaviour (what the app does)

  • User loads or selects a floor plan → app shows the Proposed plan as an interactive SVG rendered from geometry (not a PNG).
  • Delta view = computed diff of original vs proposed geometry: additions green, removals red; change register rendered like a git diff.
  • User selects rooms and walls — whole or a chunk (to remove, or add a door/window), long-press to multi-select when a change spans objects.
  • Comments accumulate in a change list → Send → agent actions them → new version → whole app updates (headline stats on top, details below — the exact look and feel of the s00 review page).

Core concept — the plan is a shape object

Everything hangs off one typed geometry model (“plan schema v2”), an evolution of today's plan_v##.json. The load step converts a plan into objects with stable ids; the app renders SVG from it; the agent edits it with validated operations; the diff engine compares two of them. Walls become first-class: derived once from room adjacency (shared rectangle edges → wall segments between junctions), then addressable.

Interaction spec

GestureResult
Click roomSelect the room polygon.
Click wallSelect the whole segment (junction → junction). Walls get a fat invisible hit-stroke (~14px) so they're easy to grab.
Drag handles on a selected wallTrim the selection to a chunk (t0–t1) — “remove this bit”, “door here”, “window here”.
Long-press (≥400 ms) then clickMulti-select mode: each click adds/removes an object (rooms and walls mix freely). Shift-click is the desktop equivalent.
With selection → type commentPopover shows the target chips; Add pushes to the change list (nothing sent yet).
SendSubmits the queued list to the agent; app streams progress and swaps in the new version when ready.
Esc / click empty spaceClear selection.

Constraint (v1)

Rooms are axis-aligned rectangles, walls are axis-aligned segments — same constraint the Pillow renderer has today. Diagonal/curved geometry is explicitly out of scope for v1 and listed under risks.

Diff engine & git-style change register

Identity first: ids are stable across versions (agent operations must reuse ids for modified objects, mint new ones for additions). Diffing original → proposed is then object set comparison: add remove modify per room, wall, and opening.

Delta view (visual)

  • Proposed plan as the base drawing.
  • Added walls/rooms: green stroke / light green fill.
  • Removed: red dashed ghosts drawn where the old geometry was.
  • Modified (repurposed room, resized): outlined with a “was → now” label, e.g. LOUNGE → BED 5.
  • Toggle: vs original / vs previous version.

Each hunk keeps the workflow's metadata: change id, title, rent impact, compliance flags — so the register stays the same “paper trail” the repo files carry today, just rendered like a diff.

Agent contract

The agent never edits SVG or pixels. It emits typed geometry operations (split_room, merge_rooms, set_kind, add_opening(wall, t0, t1, type), remove_wall_chunk, add_fixture…) that plan-core validates (no orphan openings, no overlapping rooms beyond wall tolerance, circulation reachability check) and applies. Invalid ops bounce back to the agent with the validator's reason — same self-check loop the chat workflow uses with rendered PNGs, but mechanical.

Risks, open questions, failure modes

  • Image → geometry is the hardest ML step. That's why it's Phase 4, behind a library-first app, and always passes through a human scale-confirm + approve wizard. Failure mode: bad extraction → user edits the draft or falls back to manual JSON.
  • Wall-chunk UX is fiddly. Mitigation: whole-segment select by default, chunk only via drag handles; snap t-values to 0.05; generous hit strokes.
  • Agent edits must not corrupt geometry. All ops through plan-core validation with bounce-back; versions immutable; head-only writes (a submitted batch targets the version it was drawn on — stale submits are rejected with a rebase prompt).
  • Rect-only model can't express diagonal walls, curved balconies, multi-storey links. v1 constraint, documented; multi-storey = one plan object per level, later.
  • Two renderers, one look: the d3/SVG canvas and the Pillow exporter must stay visually in sync — plan-core owns the shared style constants (wall thickness, fills, label rules incl. the “dims on every room” rule) so neither drifts.
  • Open question: auth provider at P6 (Cognito vs Entra External ID like data-qa-agent). Deliberately deferred — local app has no auth.
