02 The root cause behind four complaints — the pending-object lockout a deliberate trade-off that must now be fixed properly

When you add or duplicate anything, the op batches locally and the canvas draws a preview object with a temporary id (pv-room1). Yesterday’s 500-error fix made those preview objects unselectable — the safe-but-blunt way to stop edits referencing ids the server never sees. That is exactly why new rooms are stuck, copies can’t be moved, new fixtures can’t be labelled. The proper fix: edits on a pending object rewrite its pending op in place — drag the copy, the queued add_room’s x/y updates; rename it, the op’s name updates. No pv-id ever reaches the server (the server-side guard stays as a backstop).

04 Clear dimensions — one honest number walls subtracted; verified against live geometry

Room rects span to the wall centreline gap, so today’s labels overstate usable space — and some rooms still carry a third, hand-authored number from the source file. The standard becomes: the label is the clear internal size — rect span minus each side’s wall encroachment (wall inner face → room edge). Live worked example, BED 3: encroachment left 0.04 · right 0.11 · top 0.11 · bottom 0.04 → 3.00 × 3.00 rect ⇒ 2.85 × 2.85m clear.

RuleDetail
Formulaclear = rect span − Σ per-side max(0, wall inner face − room edge), computed from the derived walls (interior half 0.09m, exterior 0.11m). Implemented once in geometry.ts + mirrored in plan-core so register/describe/export agree.
Canvas labelsShow clear size (decision Q2 covers whether rect shows too). Authored dims strings from source files are retired — no more “3.1 × 3.1”.
Inspectorx/y/w/h fields stay in rect space (that’s op space) with a live “≈ 2.85 × 2.85m clear” readout beside them.
Register / export / areasdescribe() lines, PNG export labels, and the m² figures in the context bar all switch to clear numbers.
ContractFeature check C2: every room’s clear size ≥ 0; BED 3 fixture case asserts 2.85 × 2.85 ± 0.01.

07 Risks & guard-rails

Wall-move ambiguity Moving a shared wall changes two rooms at once; a mis-drag ricochets. Guard: live dim labels on both rooms during the drag, 0.05m snap, min-size clamps, and ⌘Z. Exterior walls are simply immovable (envelope contract).
Pending-op rewrite complexity Multiple edits to one pending object must merge into one op (not stack). Guard: the pending list keys ops by preview id; W5’s parity check (preview == server result) makes drift visible immediately.
Clear dims near openings An opening doesn’t reduce wall thickness for dimensioning — clear size uses the wall face regardless. Stated in the spec so the numbers stay stable when doors are added.
Sandbox hygiene Crashed check runs could leak sandbox reviews. Guard: sandboxes are flagged, hidden from the library, and swept on backend start + before each run.
Matrix drift New canvas features that skip the registry silently escape testing. Guard: PR convention — any change under PlanCanvas/Inspector/editing.ts must touch features.ts; the Feature Checks tab shows a “features without checks” count.
