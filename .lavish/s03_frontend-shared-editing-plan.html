02 Target architecture — one op pipeline, two authors how “everything a human or AI can control” actually works

The backend already has the perfect seam: apply_ops() in plan-core/ops.py is “the ONLY way the agent may edit a plan” — typed, validated, envelope-safe. The plan is not to build a second editing system. Direct human edits compile to the same op list and enter through a new no-LLM endpoint. Same validator, same wall re-derivation, same register diff, same version history — hunks just gain an author field.

07 Risks & failure modes

Version explosion from micro-edits Ten drags = ten versions would trash the register’s narrative. Mitigation (decided, Q1): pending-ops preview batches everything until one Apply; one Apply = one version = one hunk.
Rent integrity Agent changes carry a rent estimate; human edits don’t. If a human deletes a bedroom, proposed rent is silently wrong. Mitigation (decided, Q2): zero-impact hunk flagged “rent not re-assessed”, with a one-click “re-estimate” that asks the agent.
Concurrent writers A human Apply while an agent job runs would fork the head. Mitigation: single-writer lock — /edits 409s while a job is queued/running (the UI already tracks busy), and the existing stale-head 409 covers the reverse case.
Fixture id backfill Old versions store fixtures without ids. Assign deterministic ids on read (fx:label-slug-i by array order) so diffs between old and new versions stay stable; never mutate stored v≤head JSON in place.
Scope note — walls stay derived Humans move rooms, not walls: derive_walls() stays the single source of wall truth, which is what keeps the envelope invariant enforceable. The UI should say this (“walls follow rooms”) rather than let users try to drag a wall sideways and wonder why they can’t.
